为什么引用既有jar包，又有aar？

  • jar，全称为Java Archive，即Java 归档文件，可以用于发布和使用类库（当然也有其他用途）。
  • aar 文件是在jar基础上开发的，它之所以开发出来是因为一些android特有的文件需要被包含进去，比如AndroidManifest.xml、 Resources、Assets 或者 JNI，而这些文件是不包含在jar标准里的。

    简单来说，aar 与 jar都是 zip压缩文件，只是文件结构不同。jar 文件被集成在 aar 里，以 classes.jar命名。

Java中堆栈及内存分配

Posted on 2018-05-14 | Edited on 2018-05-15 | In Java

写项目时，内存溢出、OutOfMemoryError、StackOverflowError 是很常见的bug。字面意思很明白，比如某个对象存储需要100M，但栈空间只剩下80M，这时程序就会抛出内存溢出之类的异常。解决措施也很多，比如变量使用后将其设为null，或者调用System.gc() 。说实话这些解决方式使用时有些盲目，我觉得理解清楚堆栈以及内存分配是解决这类问题的第一步。

首先申明这篇文章我不是首创，只是觉得原作者很好地解释了我的疑问。这里我做下记录（原文章是英文，文章最后我会配上链接），希望可以帮助更多人理解它们之间的关系。

堆(Java Heap Space）

Java堆空间由java运行时使用，用来给对象和类分配内存。任何java对象，都是在堆空间里被创建。

垃圾回收机制在栈中运行，用来释放栈中没被引用的对象所占的内存。在堆空间创建的对象全局可访，并且在应用的任何地方都可被引用。

栈(Java Stack Memory)

栈用来运行一个线程，栈内会含有一个方法的局部变量和引用变量。栈内存遵循后进先出（Last-In-First-Out） 原则，当一个方法被调用，一个新的区块就在栈里创建了，用来存储该方法所使用的局部变量和引用。方法调用结束后，该区块废弃了，并且可被下一个方法使用。

栈的空间大小远小于堆。

程序中的堆栈

下面用一段简单的程序来理解下堆栈内存的使用

下面这张图展示了与上述程序相关的堆栈内存，以及它们是如何存储原始数据类型、对象以及引用变量的。

  • 一旦程序运行起来，所有运行时的类都被加载进堆空间。执行到main()（Line 1）方法，java 运行时会创建栈内存，供main()方法所在的线程使用。
  • Line 2，我们在main()方法的栈内存里生成，并存储了原始数据类型的局部变量。
  • Line 3，因为我们创建了对象，所以它被存储在堆空间。与此同时，栈里也包含了该对象的引用。
    Line 4，我们创建了Memory对象，该对象也是被存储于堆空间，同时栈里也包含该对象的引用。
  • 在Line 5，我们调用了foo()方法。此时会在栈顶创建一个新的区块，用来被foo()方法所在的线程使用。同时因为传入了Object对象，foo()对应的栈空间会创建Object对象的引用（Line 6）。
  • Line 7，字符串被创建，该字符串进入String Pool。同时，foo()对应的栈空间会创建该字符串的引用。
  • Line 8，foo()方法执行完毕。此时，栈里分配给该方法所在线程的内存会被释放掉。
  • Line 9，main()方法执行完毕，此时它所占有的栈内存被释放。同样地，整个程序也在此行结束，java运行时释放所有占用的内存，并结束程序的运行。

堆栈间的关系

基于上述的逐条说明，我们可以得出以下结论：

  1. 堆空间可以被整个应用使用，然而栈内存只能被一个正在执行的线程使用。
  2. 不管什么时候创建对象，它总是被存储在堆空间里，栈只会含有它的引用。栈内存只会含有原始数据类型的局部变量，以及对堆里对象的引用。
  3. 栈里的对象都是全局可访问的，然而对空间不能被其他线程访问。
  4. 栈的内存管理满足后进先出院子，而堆空间因为全局可用，管理起来会复杂的多。堆空间可以划分为 Young-Generation，Old-Generation 等等。具体参见垃圾回收机制。
  5. 栈内存生命周期很短，线程执行结束其对应的栈空间也就释放掉了。而堆空间从程序开始运行到结束，会一直存在。
  6. 我们可以使用-Xms和-Xmx这两个jvm选项定义堆内存的区间，使用-Xss定义栈内存。
  7. 栈空间不足，java 运行时会抛出java.lang.StackOverFlowError异常。堆空间不足，则抛出java.lang.OutOfMemoryError: Java Heap Space异常。
  8. 栈内存比堆内存小得多。由于内存分配简单(LIFO)，栈的运行速度比堆更快。

加密算法和分组密码

Posted on 2018-05-05 | Edited on 2018-05-07 | In PKI

一直没搞懂对称加密算法与分组密码的关系，趁着周六，重新梳理了这两个概念，现用自己的话将两者的关系表述清楚。

对称加密算法

对称加密算法是指加密和解密使用同一个秘钥，具体定义可以参考之前的文章加密算法。

分组密码

分组密码是指将明文消息编码表示后的数字（明文数字）序列，划分为n组，然后在秘钥的控制下变换成等长的输出数字（密文数字）序列。

注意：分组密码定义的后半句，在秘钥的控制下变换成等长的输出数字序列，这种变换所使用的算法，可能就是对称加密算法。

此时，算法和分组之间的关系已经明了。一句话概括为，先分组，再对每组施以算法。

在分组加密的基础上，又有了加密模式和填充方式这两个概念。接下来，我想简单聊聊这两个概念，最后会推荐两篇别人的文章以加深理解。

加密模式

记得刚加入芯片管家团队时，对加密模式的作用不太理解。同事解释道，不同的加密模式，使明文被破解的难度不同。例如，CBC比ECB更难被破解，因为在CBC模式下，加密之前，会将明文块与之前的密文块异或，再对异或结果进行加密。

除了CBC和ECB，还有CFB和OFB模式。这篇文章，对常见的4种模式作了很好地阐述。

填充方式

某些解密模式（即CBC和ECB），当最后一组的长度不足分组的固定大小，就要对最后一组进行填充。有的规则要求全部填充为0，有的要求第一位填1，其他位填0。

加密算法

Posted on 2018-04-23 | Edited on 2018-04-25 | In PKI

本人所在公司是信息安全型公司，大部分业务都是以加密算法为基础，在此对常用加密算法进行了总结。

对称加密算法

加密和解密使用相同秘钥，这是对称加密算法。它的特点是简单快速，通常使用较小的秘钥，一般小于256bit，因为秘钥越长，加解密越耗时。

常见的算法有SM1、SM4、AES128等。这些算法都采用对称分组密码体制,秘钥长度和分组长度均为128 bit，即16 byte。对消息进行加解密时，若消息过长，则需要分组；若消息过短，则需要填充。

非对称算法

与对称加密算法不同，非对称算法使用一对秘钥，即公钥（public key）和私钥（private key）。私钥由安全一方保管，不能外泄。非对称加密算法使用其中一个加密，使用另一个解密。

常见的算法有SM2、RSA。SM2秘钥长度是256,RSA秘钥长度是1024或者2048。SM2算法基于ECC（椭圆曲线公钥算法），相较于RSA，SM2安全强度更高，计算速度更快。

摘要算法

摘要算法，加密过程不需要秘钥且无法解密，相同的明文数据经过相同的消息加密算法才能得到相同的密文。

主要特征：

  • 无论明文多长，计算出来的摘要长度是固定的。摘要越长，越安全。

    • MD5产生128个比特位的摘要消息。
    • SHA-1可以产生160比特位的摘要消息，SHA-1的变体可以产生192和256比特位的摘要消息。
    • SM3产生256位（32byte）摘要消息。

  • 相同明文，相同算法，产生相同摘要消息。相反的，不同明文，相同算法，产生不同摘要消息。

常用于数字签名。

数字签名

Posted on 2018-03-29 | Edited on 2018-04-25 | In PKI

公司项目里涉及数据加解密，不了解相关基础知识，做起项目就像盲人摸象。周三下午不太忙，趁着这段时间，将项目中涉及的概念梳理了一遍。

证书中心（CA）

CA，即Certificate Authority。作为权威的、公正的、可信赖的第三方机构，专门负责发放并管理所有网上交易的实体所需的数字证书。

数字签名

数字签名，又称公钥数字签名。只有信息的发送者才能产生，别人无法伪造的一段数字串，这段数字串也是对信息的发送者发送信息的真实性的一个有效证明。

举个栗子：

  • 发送方：发送报文时，先使用一个哈希函数生成报文摘要，再用发送者的私钥对摘要加密生成加密摘要，最后将加密摘要和报文一起发送出去。该过程称为签名。
  • 接收方：接收到消息后，先用和发送方一样的哈希函数对原始报文生成摘要，然后使用发送方公钥对接收的加密报文解密。若两者相同，则可以确认数字签名是发送方的。该过程称为验签。

为什么对报文摘要加密，而不是对原报文加密？

  • 因为RSA加解密非常耗时，被加密的报文越大，耗时越多。而报文摘要（同一个报文通过哈希算法生成相同的摘要，不同报文生成的摘要不同）要比原始报文小的多，这样加解密耗时少。

数字证书

数字证书是一个经过证书授权中心CA数字签名的包含公开秘钥拥有者和公开秘钥的文件。

具体过程如下：

  1. 用户产生自身公私钥
  2. 用户将公钥和部分个人身份信息发送给认证中心
  3. 认证中心经过必要步骤确认用户身份
  4. 确认通过，颁发数字证书给用户。该证书内包含公钥、身份信息，以及认证中心的签名信息。
